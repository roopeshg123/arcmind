 PART 1 — WHAT IS THIS PROJECT?
════════════════════════════════════════════════════════════════════════════════

This project is an AI-powered assistant that reads your Arc application’s
HTML documentation, Jira ticket history, and Confluence wiki pages, then
answers questions through a chat interface.

Think of it as a private ChatGPT that ONLY knows about YOUR application
and YOUR team’s internal knowledge.

Instead of searching through hundreds of HTML files manually, you type a
question in natural language and instantly get an accurate, sourced answer.

Example:
  You ask  : "How do I configure SFTP authentication in Arc?"
  AI replies: "To configure SFTP authentication, navigate to the SFTP connector
               settings... [answer from your actual documentation]"
               Source: SFTP.html, SFTP-Server.html

This is possible because of a technology called RAG (explained in Part 3).

 PART 2 — WHAT IS ARTIFICIAL INTELLIGENCE & LLM?
════════════════════════════════════════════════════════════════════════════════

──────────────────────────────────────────
 2.1  What is Artificial Intelligence (AI)?
──────────────────────────────────────────

Artificial Intelligence is the ability of a computer program to perform tasks
that normally require human intelligence — such as:
  - Understanding written language
  - Answering questions
  - Summarising documents
  - Writing code
  - Translating languages

AI is not magic. It is a mathematical model trained on vast amounts of text
data that has learned patterns of language.


──────────────────────────────────────────
 2.2  What is an LLM (Large Language Model)?
──────────────────────────────────────────

An LLM (Large Language Model) is the specific type of AI used in this project.

Definition:
  An LLM is a deep learning model trained on billions of text documents
  (websites, books, articles, code) to predict and generate human-like text.

How it works (simplified):
  1. During training, the model reads billions of sentences.
  2. It learns: "given these words, what word comes next?"
  3. Over trillions of such predictions, it learns grammar, facts, reasoning,
     and how to answer questions.
  4. The result is a model with hundreds of billions of internal parameters
     (numbers) that encode all this learned knowledge.

When you ask a question:
  - The LLM does NOT search the internet.
  - It generates a response token-by-token based on its learned patterns.
  - It is predicting "what is the most useful and accurate next word?"

Popular LLMs:
  - GPT-4o, GPT-4o-mini  (OpenAI) ← used in THIS project
  - Claude 3.5 Sonnet     (Anthropic)
  - Gemini 1.5 Pro        (Google)
  - LLaMA 3               (Meta, open source)


──────────────────────────────────────────
 2.3  What are Tokens?
──────────────────────────────────────────

LLMs do not read words — they read TOKENS.

A token is roughly 3-4 characters or 0.75 words.
  "Hello world"   = 2 tokens
  "SFTP connector" = 3 tokens

Why does this matter?
  - Every LLM has a context window (maximum tokens it can read at once)
  - GPT-4o-mini handles ~128,000 tokens (~96,000 words) at once
  - This is why we split documents into chunks — to stay within limits
  - OpenAI charges per token used (both input and output)


──────────────────────────────────────────
 2.4  What is OpenAI?
──────────────────────────────────────────

OpenAI is the company that created the GPT series of LLMs.

They provide access to their models through an API (Application Programming
Interface) — meaning your code sends a request over the internet and gets
back an AI-generated response.

OpenAI API services used in this project:

  ┌─────────────────────────────┬───────────────────────────────────────────┐
  │ Service                     │ Purpose                                   │
  ├─────────────────────────────┼───────────────────────────────────────────┤
  │ Chat Completions API        │ Generates answers to your questions       │
  │ (gpt-4.1)                   │ Endpoint: /v1/chat/completions            │
  ├─────────────────────────────┼───────────────────────────────────────────┤
  │ Embeddings API              │ Converts text into 3072-dim vectors       │
  │ (text-embedding-3-large)    │ Endpoint: /v1/embeddings                  │
 2.5  What are Embeddings / Vectors?
──────────────────────────────────────────

An embedding is a way to convert text into a list of numbers (a vector) that
captures its MEANING.

Example:
  "How to configure SFTP?"     →  [0.023, -0.817, 0.442, ... 3072 numbers]
  "SFTP setup and connection"  →  [0.019, -0.801, 0.431, ... 3072 numbers]

Notice those two vectors are very close to each other — because the sentences
mean nearly the same thing, even though the words are different.

This is the key insight behind semantic search:
  You don't search for exact words. You search for similar MEANING.

That is why you can ask "How do I set up file transfer?" and still find
documentation about "SFTP connector configuration" — the meanings are close.

 PART 3 — WHAT IS RAG (RETRIEVAL-AUGMENTED GENERATION)?
════════════════════════════════════════════════════════════════════════════════

──────────────────────────────────────────
 3.1  The Problem RAG Solves
──────────────────────────────────────────

LLMs like GPT-4o-mini have a problem when used for private documentation:

  Problem 1 — Training cutoff:
    GPT was trained on internet data up to a certain date.
    It has NEVER seen your Arc application documentation.
    So if you ask it about Arc, it will make up wrong answers (hallucinate).

  Problem 2 — Context window limit:
    Even if you pasted your entire documentation into every request,
    541 HTML files × average 15,000 characters = ~8 million characters.
    That far exceeds any model's context window and would cost $$$$ per query.

  Problem 3 — Accuracy:
    LLMs sometimes "hallucinate" — confidently stating wrong facts.
    With RAG, the model is forced to answer from your actual documents.


──────────────────────────────────────────
 3.2  What is RAG?
──────────────────────────────────────────

RAG = Retrieval-Augmented Generation

It is an architecture that combines two systems:
  1. RETRIEVAL  — a search system that finds relevant document chunks
  2. GENERATION — an LLM that reads those chunks and writes an answer

The key idea: instead of asking the LLM to remember everything, you
RETRIEVE the relevant information at query-time and GIVE it to the LLM.

Analogy:
  Without RAG: You ask someone a question and they answer from memory.
  With RAG:    You ask someone a question, they first look up the answer
               in the right book, read it, then explain it to you.

The LLM becomes a "reader and explainer" rather than a "memoriser".


──────────────────────────────────────────
 3.3  The RAG Pipeline — Step by Step
──────────────────────────────────────────

INDEXING PHASE (runs once when you click ⚡ Index Docs):

  Step 1: READ
    ┌─────────────────────────────────────────┐
    │  Option A — Web crawl (DOCS_URL set):   │
    │  Starts at https://cdn.cdata.com/help/  │
    │  AZN/mft/ and follows all links within  │
    │  that URL prefix, fetching each page    │
    │  with the requests library              │
    │                                         │
    │  Option B — Local files (DOCS_DIR set): │
    │  Reads .html files from local folder    │
    │                                         │
    │  Both paths use BeautifulSoup to strip  │
    │  nav, scripts, styles and extract text  │
    └─────────────────────────────────────────┘

  Step 2: SPLIT
    ┌─────────────────────────────────────────┐
    │  Each file's text is split into chunks  │
    │   - Docs chunk size:   1500 tokens      │
    │   - Docs overlap:       300 tokens      │
    │   - Jira chunk size:    600 tokens      │
    │   - Jira overlap:       100 tokens      │
    │  540 files → ~2,800 doc chunks          │
    │  Jira: one main doc + per-comment docs  │
    └─────────────────────────────────────────┘

  Step 3: EMBED  (OpenAI Embeddings API called here)
    ┌─────────────────────────────────────────┐
    │  Each of the 4179 chunks is sent to     │
    │  OpenAI Embeddings API                  │
    │  Returns a 3072-number vector per chunk │
    │  (text-embedding-3-large)               │
    │  4179 chunks → 4179 vectors             │
    └─────────────────────────────────────────┘

  Step 4: STORE
    ┌─────────────────────────────────────────┐
    │  All vectors + original text saved to   │
    │  ChromaDB (local SQLite file)           │
    │  THREE collections:                     │
    │   arcmind_docs        (doc chunks)       │
    │   arcmind_jira        (ticket chunks)    │
    │   arcmind_confluence  (wiki page chunks) │
    │  Located at: ./chroma_db/               │
    └─────────────────────────────────────────┘


QUERY PHASE (runs every time you ask a question):

  Step 1: REWRITE (if conversation history exists)
    ┌─────────────────────────────────────────┐
    │  History: "What is REST connector?"     │
    │  You ask: "How do I authenticate it?"   │
    │                                         │
    │  LangChain sends both to GPT:           │
    │  "Rewrite as standalone question"       │
    │                                         │
    │  GPT rewrites:                          │
    │  "How do I authenticate the REST        │
    │   connector in Arc?"                    │
    └─────────────────────────────────────────┘

  Step 1b: SESSION MEMORY LOOKUP
    ┌─────────────────────────────────────────┐
    │  If session_id is provided:             │
    │  conversation_memory.py loads stored    │
    │  history for that session (in-memory,   │
    │  keyed by session_id, max 128 chars).   │
    │  History is merged with any passed-in   │
    │  chat_history for context.              │
    └─────────────────────────────────────────┘

  Step 2: CONNECTOR DETECTION + QUERY ROUTING
    ┌─────────────────────────────────────────┐
    │  connector_detector.py scans the query  │
    │  for known Arc connector / component    │
    │  names (SFTP, REST, Jira, etc.)         │
    │                                         │
    │  query_router.py decides:               │
    │   - docs-only (how-to questions)        │
    │   - jira-only (bug/ticket questions)    │
    │   - hybrid  (default for most queries)  │
    └─────────────────────────────────────────┘

  Step 3: QUERY EXPANSION
    ┌─────────────────────────────────────────┐
    │  query_expander.py sends the query to   │
    │  gpt-4.1 (singleton, cached)            │
    │  Generates 5 variants covering:         │
    │   - synonyms                            │
    │   - acronym expansions                  │
    │     (e.g. "MDN" → "Message Disposition  │
    │      Notification")                     │
    │   - related sub-topics                  │
    └─────────────────────────────────────────┘

  Step 4: HYBRID RETRIEVAL
    ┌─────────────────────────────────────────┐
    │  Each of the 5 variants is searched:    │
    │   a) ChromaDB vector search (semantic)  │
    │   b) BM25 keyword search (exact terms)  │
    │  Across THREE collections:              │
    │   arcmind_docs, arcmind_jira,           │
    │   arcmind_confluence                   │
    │  Results merged + deduplicated.         │
    │  Top RETRIEVER_TOP_K candidates kept.   │
    └─────────────────────────────────────────┘

  Step 5: RERANKING (cross-encoder, local)
    ┌─────────────────────────────────────────┐
    │  reranker.py scores each (query, chunk) │
    │  pair jointly using the cross-encoder.  │
    │  Far more accurate than cosine sim.     │
    │  Runs entirely on-device (~22MB model). │
    │  Top RERANKER_TOP_N chunks kept.        │
    └─────────────────────────────────────────┘

  Step 6: GENERATE (OpenAI Chat Completions API)
    ┌─────────────────────────────────────────┐
    │  Prompt sent to gpt-4.1:                │
    │                                         │
    │  SYSTEM: "You are ArcMind assistant.    │
    │   [Arc Documentation Context]           │
    │   [chunk 1 from REST.html]              │
    │   [chunk 2 from REST.html]              │
    │   [Confluence Wiki Context]             │
    │   [page: REST Setup Guide (with URL)]   │
    │   [Jira Issues Context]                 │
    │   [cluster: ARC-1234, ARC-5678]"        │
    │                                         │
    │  HISTORY: [previous messages]           │
    │  USER:    "How do I authenticate it?"   │
    │                                         │
    │  Confluence links rendered as clickable │
    │  hyperlinks in the chat UI.             │
    │  Streaming available via /api/chat/stream│
    └─────────────────────────────────────────┘

  Step 7: RETURN + LOG
    ┌─────────────────────────────────────────┐
    │  Answer text sent back to browser       │
    │  Source files listed below the answer   │
    │  Confluence pages cited as clickable    │
    │  hyperlinks (title + /wiki/ URL)        │
    │                                         │
    │  _log_query() appends a JSON record to  │
    │  logs/query_log.jsonl:                  │
    │   {ts, question, expanded_queries,      │
    │    docs_retrieved, jira_retrieved,      │
    │    confluence_retrieved,               │
    │    answer_preview (first 500 chars)}    │
    │                                         │
    │  session_id memory updated if provided  │
    └─────────────────────────────────────────┘

 PART 7 — HOW THE OPENAI API IS CALLED (EXACT DETAIL)
════════════════════════════════════════════════════════════════════════════════

──────────────────────────────────────────
 7.1  Embeddings API Call
──────────────────────────────────────────

Called during: ingestion (for all chunks) AND for each user question

HTTP Request:
  POST https://api.openai.com/v1/embeddings
  Authorization: Bearer sk-your-key
  Content-Type: application/json

  Body:
  {
    "model": "text-embedding-3-large",
    "input": "To configure SFTP, navigate to the connector settings..."
  }

HTTP Response:
  {
    "data": [{
      "embedding": [0.023, -0.817, 0.442, 0.198, ...],  // 1536 numbers
      "index": 0
    }],
    "usage": { "prompt_tokens": 42, "total_tokens": 42 }
  }

Cost: ~$0.00013 per 1000 tokens (text-embedding-3-large)
      Indexing ~20,000 chunks costs roughly $0.15–0.20

 7.2  Chat Completions API Call
──────────────────────────────────────────

Called during: every user question (after retrieval)

HTTP Request:
  POST https://api.openai.com/v1/chat/completions
  Authorization: Bearer sk-your-key
  Content-Type: application/json

  Body:
  {
    "model": "gpt-4.1",
    "temperature": 0.2,

        "content": "You are a helpful assistant that answers questions about

                    [Chunk 3: text from SFTP-Server.html about setup...]
                    [Chunk 4: text from SCP.html about key files...]
                    [Chunk 5: text from SFTP.html about error handling...]"

        "role": "user",

      },

 PART 8 — HOW LANGCHAIN WORKS IN THIS PROJECT
════════════════════════════════════════════════════════════════════════════════

LangChain is a framework that chains together AI components.
Without LangChain, we would manually write all the API calls, prompt
construction, history management, and retrieval logic ourselves.

LangChain components used:

  1. ChatOpenAI
     Wraps the OpenAI Chat Completions API.
     Handles: API key, model selection, temperature, streaming, retries.

  2. OpenAIEmbeddings
     Wraps the OpenAI Embeddings API.
     Handles: Batching multiple chunks into single API calls (efficient).

  3. Chroma (from langchain-chroma)
     Wraps ChromaDB.
     Provides: as_retriever() — turns vector DB into a LangChain retriever.

  4. RecursiveCharacterTextSplitter (from langchain-text-splitters)
     Splits long text into chunks.
     Strategy: tries to split on \n\n first, then \n, then ". ", then " "
     Preserves semantic boundaries as much as possible.

  5. ChatPromptTemplate
     Defines the structure of prompts sent to the LLM.
     Uses placeholders: {context}, {input}, {chat_history}

  6. create_history_aware_retriever
     Chain 1: Takes question + history → rewrites question → retrieves chunks.

  7. ContextualCompressionRetriever (new)
     Wraps the MMR retriever with a cross-encoder reranker.
     After MMR returns 30 candidates, the cross-encoder re-scores each
     (query, chunk) pair jointly and returns the top 8 most relevant.

  8. create_stuff_documents_chain
     Chain 2: Takes reranked chunks → stuffs them into prompt → LLM answers.

  9. create_retrieval_chain
     Connects Chain 1 and Chain 2 into one end-to-end pipeline.

  Chain execution flow:
  input question + history
        │
        ▼
  create_history_aware_retriever
        │ rewrites + expands question using gpt-4.1
        ▼
  Chroma MMR retriever
        │ returns 30 diverse candidate chunks (TOP_K × 2)
        ▼
  ContextualCompressionRetriever (cross-encoder reranker)
        │ re-scores all 30 → keeps top 8 (RERANKER_TOP_N)
        ▼
  create_stuff_documents_chain
        │ injects top 8 chunks into strict rules prompt
        ▼
  ChatOpenAI (gpt-4.1, temperature=0.1)
        │ generates grounded answer
        ▼
  {"answer": "...", "context": [8 Document objects]}

 PART 9 — WHY THIS APPROACH IS USEFUL
════════════════════════════════════════════════════════════════════════════════

  ✓ ACCURACY
    The LLM can only answer from your actual documentation.
    It cannot make up features that don't exist in Arc.
    Every answer is traceable to a source file.

  ✓ PRIVACY
    Your documentation never leaves your machine permanently.
    ChromaDB is local (no cloud database).
    Only the text chunks relevant to each question are sent to OpenAI.
    The full documentation is never uploaded anywhere.

  ✓ SECURE
    API-key middleware (X-API-Key) locks down all /api/* endpoints.
    CORS is configurable — restrict to known origins in production.
    Session IDs are validated (max 128 chars, safe charset).
    asyncio.Lock prevents concurrent ingest race conditions.
    BM25 pickle files are path-validated before loading (_safe_pickle_load).

  ✓ ALWAYS UP TO DATE
    Click ⚡ Index Docs whenever you update your HTML files.
    Use Incremental Jira Sync to pull only the latest tickets (last N hours).
    Use Update Confluence to sync changed Confluence pages via SHA-256 diff.
    The index rebuilds in minutes with the latest content.

  ✓ COST EFFECTIVE
    One-time indexing of 541 files costs ~$0.05 in OpenAI credits.
    Each question costs less than $0.001.
    No monthly subscriptions, no per-seat licensing.

  ✓ FAST
    After indexing, answers are generated in 2-5 seconds.
    ChromaDB vector search over 20,000+ vectors takes milliseconds.
    Cross-encoder reranker runs locally (no extra API call).
    Streaming answers via /api/chat/stream for instant first-token response.
    ChatOpenAI and expansion LLM are singletons (no repeated instantiation).

  ✓ CONTEXT-AWARE
    The AI remembers your full conversation.
    Follow-up questions ("How do I do that with authentication?")
    correctly understand what "that" refers to.

  ✓ PORTABLE — DOCKER / PODMAN SUPPORT
    Runs in a container with docker compose up --build.
    No Python install required on teammates' machines.
    Works identically on Windows, macOS, and Linux.
    Reranker model is pre-downloaded at build time (no runtime delay).

 PART 11 — GLOSSARY OF TERMS
════════════════════════════════════════════════════════════════════════════════

  Term                  Definition
  ────────────────────────────────────────────────────────────────────────────
  AI                    Artificial Intelligence — software that performs
                        tasks requiring human-like reasoning

  LLM                   Large Language Model — an AI model trained on text
                        to understand and generate human language

  GPT                   Generative Pre-trained Transformer — OpenAI's LLM
                        family (GPT-4, GPT-4o, GPT-4.1)

  Token                 A unit of text (~3-4 chars). LLMs process tokens,
                        not words. Costs and limits are measured in tokens.

  Embedding             A vector (list of numbers) representing the meaning
                        of text. Similar text = similar vectors.

  Vector                A list of floating-point numbers.
                        e.g. [0.023, -0.817, 0.442, ...]

  Vector Database       A database optimised for storing and searching vectors
                        by similarity. ChromaDB is one example.

  Semantic Search       Finding content by meaning rather than exact keywords

  RAG                   Retrieval-Augmented Generation — combining a search
                        system with an LLM to answer questions from documents

  Chunking              Splitting large documents into smaller pieces so they
                        fit within LLM context windows

  Hallucination         When an LLM confidently states something incorrect.
                        RAG reduces this by grounding answers in real docs.

  Context Window        Maximum amount of text an LLM can process at once

  MMR                   Maximal Marginal Relevance — retrieval algorithm that
                        returns relevant AND diverse results (not duplicates)

  ASGI                  Async Server Gateway Interface — Python web server
                        standard. Uvicorn is an ASGI server.

  API                   Application Programming Interface — a way for two
                        programs to communicate with each other

  REST API              A type of API that uses HTTP requests (GET, POST, etc.)

  FastAPI               Python web framework for building REST APIs

  ChromaDB              Open-source local vector database

  LangChain             Framework for building LLM-powered applications

  Cross-Encoder       A model that reads a (query, document) pair together
                      and scores their relevance. More accurate than embedding
                      similarity but slower. Used in the reranking step.

  Reranking           A second-pass relevance scoring step. MMR retrieves a
                      broad candidate set; the cross-encoder reranks it to
                      surface the truly most relevant chunks for the LLM.

  Temperature         LLM parameter controlling randomness.
                      0.0 = deterministic/factual, 1.0 = creative/random
                      Set to 0.1 in this project for maximum factual accuracy.

  Embeddings API        OpenAI service that converts text to vectors

  Chat Completions API  OpenAI service that generates text responses
